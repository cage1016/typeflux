macOS 语音输入法（Swift）需求规格说明书（更新版 Draft）

1. 背景

在 macOS 上实现一款“按住说话、松开即输入”的语音输入法工具，常驻菜单栏，通过快捷键触发录音并将语音转文字后直接插入当前光标位置，同时同步到剪贴板；提供可配置的 LLM 服务接入（OpenAI 兼容协议）、历史记录管理与导出能力，并支持“选中文本后语音指令编辑替换”的增强功能。

2. 目标
	•	极简交互：按住触发 → 录音 → 松开结束 → 流式转写/生成 → 自动插入与复制。
	•	两种模式：
	•	普通输入模式：语音转文字后插入光标位置。
	•	文本编辑模式：选中文本后触发，结合语音指令生成新文本并替换原选区。
	•	管理界面可配置快捷键与 LLM 连接信息，快捷键设置立即生效。
	•	历史记录保存最近一周，支持播放语音、导出与清空。

3. 范围

3.1 包含
	•	菜单栏常驻与状态管理
	•	快捷键触发（默认 Fn；支持配置其他组合键）
	•	录音提示浮层（固定位置、波形与计时）
	•	OpenAI 兼容协议（ChatGPT / OpenAI-compatible）对接，支持 Streaming 与失败重试
	•	文本注入（插入光标/替换选区）与剪贴板同步
	•	设置/管理界面：快捷键、LLM 配置
	•	历史记录：近一周、播放、导出、清空
	•	工程结构与编码规范

3.2 不包含
	•	历史记录的加密存储（明确暂不考虑）
	•	账号系统与跨设备同步（未提出）
	•	复杂的多语言/标点策略（未提出）

4. 功能性需求

4.1 核心交互流程

FR-1 菜单栏常驻
	•	应用启动后常驻系统菜单栏，提供入口（设置、历史、导出/清空等）及当前状态展示（就绪/录音中/转写中/失败等）。

FR-2 快捷键触发（默认 Fn + 可配置组合键）
	•	默认触发键为 Fn（按住触发）。
	•	允许在管理界面配置为其他快捷键（含组合键），用于替代/补充 Fn 触发。
	•	快捷键配置 立即生效（无需重启或额外点击“保存”才能生效）。

FR-3 按住开始录音
	•	用户按住触发键时：
	•	在屏幕正中央下方显示提示浮层；
	•	开始录音并实时显示音量波形与计时。

FR-4 松开结束录音
	•	用户松开触发键即视为输入完成：
	•	立即停止录音；
	•	进入转写/生成流程（Streaming）。

⸻

4.2 提示浮层（UI/状态）

FR-5 浮层位置与交互
	•	浮层不可拖拽，固定显示在“屏幕正中央下方”。
	•	浮层至少包含：状态文本、音量波形、计时。

FR-6 状态提示文案
	•	浮层应能显示以下状态（至少）：
	•	“正在输入中”（录音中）
	•	“转写中”（处理中）
	•	“失败”（失败态）
	•	“重复”（重复态，含义需按产品定义，见未决问题）

FR-7 失败提示与剪贴板保障
	•	当处理失败时：
	•	必须在浮层明确提示用户“失败”；
	•	仍需提供“复制到剪贴板”的可用路径（复制内容与策略见未决问题/假设）。

⸻

4.3 语音处理与接口协议

FR-8 OpenAI 兼容协议（ChatGPT/OpenAI-compatible）
	•	LLM 调用采用 ChatGPT 的 OpenAI 兼容协议。
	•	管理界面支持配置：接口地址（Base URL/Endpoint）、API Key、模型标识等必要参数。

FR-9 流式传输（Streaming）
	•	处理阶段采用 Streaming 方式获取结果（逐步返回文本）。

FR-10 重试机制
	•	调用失败时自动重试，最多 3 次（包含首次失败后的重试次数累计上限）。

⸻

4.4 文本分发（插入与剪贴板）

FR-11 自动插入
	•	转换/生成后的文本应稳稳插入到当前焦点应用的光标位置或输入点。

FR-12 剪贴板同步
	•	转换/生成后的文本需同步复制到系统剪贴板。

FR-13 全应用兼容性
	•	文本插入能力需兼容所有应用（作为强约束目标）。
	•	若出现无法注入的场景，应至少保证剪贴板同步可用，并通过 UI 反馈（与失败/降级策略一并在技术方案中固化）。

⸻

4.5 文本编辑模式

FR-14 进入编辑模式
	•	用户先选中一段文本，再按住触发键，则进入编辑模式。

FR-15 编辑逻辑
	•	选中的文本作为修改对象，结合用户语音指令进行处理，得到修改后的内容。

FR-16 自动替换与剪贴板同步
	•	修改后的内容自动替换原选区文本，并同步复制到剪贴板。

FR-17 编辑模式 UI 约束
	•	不需要特别标识“编辑模式”（不额外展示模式标签/特殊提示）。

⸻

4.6 设置与管理界面

FR-18 入口
	•	菜单栏可打开管理/设置界面。

FR-19 快捷键配置能力
	•	管理界面允许配置用于触发录音的快捷键集合（支持组合键）。
	•	配置变更 立即生效。

FR-20 LLM 配置能力
	•	管理界面允许配置 LLM 类型/模型、接口地址、API Key 等信息。

注：管理界面“可设置使用哪些快捷键”的具体可选范围/规则需在产品层面明确（见未决问题）。

⸻

4.7 历史记录管理

FR-21 历史列表
	•	提供历史记录页面，展示用户历史输入。

FR-22 保存周期
	•	仅保留最近一周的记录（按时间滚动淘汰）。

FR-23 记录内容
	•	每条记录包含：
	•	原始语音文件（支持播放）
	•	对应文本内容（最终文本）

FR-24 导出与清空
	•	历史记录页面支持：
	•	一键导出（导出范围/格式待定）
	•	一键清空（删除本地历史记录与对应音频文件）

⸻

4.8 音频文件要求

FR-25 音频格式
	•	历史存储的语音文件格式建议采用 MP3，以在体积与音质间取得平衡。

5. 非功能性需求

5.1 编码规范（强制）

NFR-1 模块划分清晰
	•	代码按领域拆分模块，例如：热键监听、音频采集、UI 浮层、LLM 客户端、文本注入、剪贴板、历史存储与导出等。

NFR-2 单一职责
	•	每个文件与类遵循单一职责原则（SRP），边界清晰、低耦合。

NFR-3 注释与可维护性
	•	对关键逻辑（系统事件监听、输入注入、音频处理链路、Streaming 解析、重试与错误处理）提供必要注释。

5.2 安全与隐私（本期约束）

NFR-4 不考虑加密
	•	历史记录与音频文件本期暂不考虑加密方案（可在后续版本扩展）。

5.3 可靠性与兼容性（强约束倾向）

NFR-5 全应用兼容
	•	“兼容所有应用”作为高优先级目标；需在技术方案中明确覆盖策略与降级路径，确保最差情况下仍可通过剪贴板完成输入。

6. 假设
	•	macOS 环境允许实现全局快捷键监听（包含 Fn 或其替代组合键）、音频录制、浮层展示、以及面向“所有应用”的文本注入能力；如系统限制导致少数场景无法注入，则按降级策略处理（至少剪贴板可用）。
	•	OpenAI 兼容协议的服务端可用且可配置（Base URL、Key、模型等）。
	•	MP3 编码能力可在本地实现或通过可接受的方式生成（具体技术实现由技术方案决定）。
